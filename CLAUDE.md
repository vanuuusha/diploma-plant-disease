## Правила написания текста

1. **Всегда** перечитай `СТИЛЬ_НАПИСАНИЯ.md` перед написанием текста — там подробные инструкции по стилю, тону, лексике, шаблонам фраз
2. Стиль — академический научно-технический, безличный, без разговорных элементов
3. Терминология — двуязычная (русский + английский в скобках при первом упоминании)
4. Опирайся на контекст из НИР1 и НИР2 — дипломная работа является их логическим продолжением
5. При описании экспериментов — конкретные числа, метрики, код (стиль НИР2)
6. При теоретических обзорах — развёрнутый анализ с преимуществами/недостатками (стиль НИР1)
7. **Никогда** не ссылаться на НИР1/НИР2 в тексте диплома (формулировки вида «что было подтверждено в НИР1», «как показано в рамках НИР2»). НИРы — это внутренний контекст для Claude, а не часть дипломной работы. Диплом — самостоятельный документ.
8. **Не писать в тексте диплома чрезмерно-технических реализационных деталей**, которые уместны в README или отчёте об имплементации, но не в научной работе. К таким деталям относятся, в частности:
   - размеры конкретных файлов в МБ/ГБ (например, «JSON-файл объёмом 84 МБ»);
   - внутренние идентификаторы, имена полей/ключей/переменных из кода (например, `task_id`, `BboxParams`, `min_visibility=0.1`, `max_detections_per_image=100`);
   - описания проблем сборки данных, не документированных особенностей экспорта, трудностей восстановления соответствия между файлами — это инженерная рефлексия, не научный результат;
   - абсолютные пути (`/code/data/dataset_final`), упоминания симлинков, имена yaml-конфигов, внутренние директории (`code/results/...`);
   - **любые текстовые ссылки на локальные файлы и директории кодовой базы** в теле диплома: формулировки вида «индивидуальные снимки содержатся в директории `code/results/task_03/examples/`», «результаты сохранены в `code/results/...`», «скрипт `code/scripts/...`», «см. файл `code/tasks/task_NN/RESULT.md`». Читатель диплома не имеет доступа к репозиторию, такие ссылки бессмысленны и разрушают целостность научного текста. В диплом идут только концепции, результаты (в виде таблиц/рисунков/чисел в самом тексте) и ссылки на опубликованные источники. Если нужно описать содержимое дополнительной директории — включить его в сам текст или в приложение. Исключение: markdown-синтаксис вставки изображений `![alt](../code/results/...)` — это рендеринговая ссылка, а не текст, при экспорте в docx/PDF она превращается в картинку и допустима;
   - точные версии библиотек («Albumentations версии 2.0.8»), если версия не является существенным элементом воспроизводимости метода;
   - упоминания форматов весов (`float16`) и объёмов видеопамяти (если только это не принципиально для метода);
   - **избыточно-скрупулёзные разъяснения внутренней механики разметки и экспорта**, не относящиеся к сути работы: «в JSON всего 18 меток, но только 9 имеют свои директории, остальные представлены как со-аннотации на чужих кадрах», «метка X есть в экспорте, но не имеет собственных снимков», «некоторые полигоны размечены на кадрах с иной преобладающей патологией» и аналогичные рассказы про устройство разметки. В научном тексте достаточно описать итоговый состав целевых классов и принцип фильтрации; детали меток, которые отсеиваются, не являются научным результатом и не нужны читателю.

Вместо этого диплом формулирует **сущность** (какие характеристики у данных, какие механизмы обеспечивают корректность, какой алгоритм применяется) на уровне концепций, параметров и результатов. Допускаются: библиотека в целом, её возможности, ключевые параметры метода (число шагов, коэффициент `strength`), численные характеристики датасета, короткие кодовые фрагменты для иллюстрации смысла (2–3 строки).

## Правила оформления

- Ссылки на рисунки: «на рисунке N» (до рисунка), подпись: «Рисунок N — Название»
- Ссылки на таблицы: «в таблице N», подпись: «Таблица N — Название»
- Кавычки — ёлочки « »
- При первом упоминании аббревиатуры — полная расшифровка + аббревиатура в скобках
- **Формулы — в LaTeX-синтаксисе:** инлайн `$formula$`, блочные `$$formula$$`. Пример: $F'_l = \gamma_l \odot F_l + \beta_l$
- **Формулы-уравнения всегда на отдельной строке** — если формула представляет собой самостоятельное уравнение или определение, она оформляется блочно (`$$...$$`) на отдельной строке, а не инлайн. Инлайн `$...$` используется только для упоминания переменных и коротких выражений внутри предложения.
- **Сноски-аннотации** — при упоминании нетривиального термина или явления (например, «проблема исчезающих градиентов», «mode collapse», «катастрофическое забывание») сразу после него в квадратных скобках даётся краткое пояснение (1–2 предложения), которое автор позже оформит как сноску в .docx. Формат: `[Краткое пояснение термина]`. Пример: «проблема исчезающих градиентов [При обратном распространении ошибки через большое число слоёв градиенты многократно умножаются на малые величины и экспоненциально затухают, что делает обновление весов ранних слоёв практически невозможным]».
- **Перечисления оформляются списками, а не сплошной прозой.** Любая конструкция вида «первая … вторая … третья …», «во-первых … во-вторых … в-третьих …», «три задачи … N этапов … K преимуществ …» — выносится в нумерованный (или маркированный) список, каждый пункт с коротким заголовком жирным шрифтом и пояснением. Это повышает читаемость и визуально разбивает текст. Сплошной абзац с «Первая — …, Вторая — …, Третья — …» считается плохим оформлением и подлежит рефакторингу.
- Подробные правила оформления: `оформление/recomendations.md` и `оформление/Чек-лист.md`

### Рисунки в тексте

Рисунки вставляются в текст в следующем формате:

```
Рисунок N — Подпись к рисунку. (Описание для вставки: краткое описание того, какую картинку нужно вставить + промпт для генерации в NanoBanano)
```

Правила:
- Рисунки нужно вставлять в местах, где это уместно — например, при описании архитектуры CNN, схемы детектора, пайплайна обработки и т.д.
- Перед рисунком в тексте обязательно ссылка: «на рисунке N показана...», «архитектура представлена на рисунке N»
- В скобках после подписи — описание-подсказка для автора: что именно изобразить и промпт для генерации через NanoBanano
- Рисунки из статей (архитектуры моделей, схемы, диаграммы) значительно повышают качество обзорной главы

### Насыщенность иллюстративным материалом

Текст дипломной работы должен быть хорошо разбавлен иллюстрациями, формулами, таблицами и диаграммами. Это важно как для наглядности, так и для объёма (количества страниц). Ориентиры:
- В обзорных главах — рисунок на каждую ключевую архитектуру/метод (схема архитектуры, блок-диаграмма, пайплайн)
- Формулы выносить в блочный формат при любой возможности — они визуально разбивают текст и добавляют научности
- Таблицы сравнения — везде, где сопоставляются 3+ объекта (модели, методы, датасеты)
- Диаграммы пайплайнов, блок-схемы алгоритмов — при описании любых процессов и архитектур

### Система задач (tasks)

Проект использует два окружения:
- **Cowork (этот сеанс)** — написание текста диплома, анализ результатов, планирование
- **Claude Code на Linux-сервере** — выполнение тяжёлых задач (обучение моделей, эксперименты, тесты)

#### Как это работает:

1. **Cowork создаёт задачу** — файл `code/tasks/task_NN/TASK.md` с описанием того, что нужно сделать.
2. **Claude Code на сервере** читает `TASK.md`, выполняет задачу, пишет результат в `RESULT.md` и пушит на git.
3. **Пользователь пишет в Cowork:** «готово task_17» — значит задача выполнена.
4. **Cowork читает** `code/tasks/task_17/RESULT.md`, анализирует результаты и использует их в тексте диплома.

#### Формат TASK.md:
```markdown
# Task NN: Краткое название

## Статус
pending | in_progress | done

## Цель
Что нужно сделать и зачем.

## Шаги
1. ...
2. ...

## Входные данные
Где лежат данные, конфиги.

## Ожидаемый результат
Куда записать результат, какие метрики/артефакты сохранить.

## Результат записать в
`code/tasks/task_NN/RESULT.md`
```

#### Формат RESULT.md:
```markdown
# Result: Task NN

## Статус
done | failed | partial

## Что было сделано
Краткое описание выполненных действий.

## Результаты
Метрики, пути к артефактам, графикам.

## Проблемы / Замечания
Если что-то пошло не так.

## Артефакты
- `code/results/task_NN/...` — путь к сохранённым файлам
```
