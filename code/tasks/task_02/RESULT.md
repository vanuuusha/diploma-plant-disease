## 1. Постановка задачи и её место в работе

Первый этап любого supervised-пайплайна детекции — приведение разметки к формату, ожидаемому выбранным детектором. В дипломной работе в качестве базового детектора предполагается семейство YOLO (решение принято по итогам НИР2: YOLOv10 показал оптимальный баланс скорости и точности среди Faster R-CNN, DETR, RT-DETR). Поэтому все аннотации нужно привести к YOLO-формату:

```
<class_id> <x_center> <y_center> <width> <height>    # все координаты нормализованы [0, 1]
```

Исходные данные — экспорт из Label Studio в JSON. Разметка выполнена в виде **полигонов** (многоугольников, описывающих контур симптома), что естественно для болезней растений: симптомы имеют произвольную форму. Однако YOLO работает с axis-aligned bounding boxes — описанными прямоугольниками. Конвертация «полигон → bbox» — шаг с потерей информации (мы теряем точную форму и сохраняем только её охватывающий прямоугольник), но это стандартная практика в задачах детекции, если не планируется instance segmentation.

## 3. Ключевое проектное решение: фильтр `MIN_IMAGES = 100`

**Что сделано**: класс сохраняется в конечном датасете только если в датасете есть ≥ 100 изображений, содержащих хотя бы одну его аннотацию.

**Почему фильтр по изображениям, а не по аннотациям**: один снимок может содержать несколько bbox одного класса (например, несколько пятен мучнистой росы на листе). Если фильтровать по аннотациям, класс с 150 полигонами на 5 снимках формально пройдёт порог, но реально при обучении модель увидит всего 5 различных кадров и переобучится на них. Фильтр по числу уникальных снимков-носителей отражает **реальное визуальное разнообразие**, доступное модели.

**Почему 100**: эмпирический порог, ниже которого обучение детектора становится статистически ненадёжным (типичный рекомендованный минимум для fine-tuning предобученного YOLO — 100–200 изображений на класс; ниже — высокий риск переобучения даже при агрессивной аугментации). 100 — компромисс: не слишком жёсткий (не теряем умеренно-редкие классы), достаточно мягкий (отсекаем однозначно нетренируемые).

**Результат фильтра**: остаётся 9 классов (id 0–8). 9 отброшенных описаны в секции «Замечания».

**Альтернативы, от которых отказались**:
- `MIN_ANNOTATIONS ≥ 100` — класс с 259 аннотациями на 7 снимках (Чернь колоса) прошёл бы, что неадекватно.
- `MIN_ANNOTATIONS ≥ 500` — даёт те же 9 классов в случае этого датасета, но критерий менее интерпретируемый (зависит от плотности bbox на кадре, которая сама по себе информативный признак, а не помеха).
- Без фильтра — 18 классов с крайне неравномерной представленностью сделали бы дисбаланс 4552× (vs 1) и обучение невозможным без специальных приёмов (focal loss, class-balanced loss).

## 4. Алгоритм конвертации полигон → bbox

Для каждого полигона, заданного списком точек в относительных координатах (%) к размерам изображения:
1. Извлекаем min/max по x и y.
2. Нормализуем в [0, 1]: делим на 100 (у Label Studio точки уже в процентах).
3. Формируем bbox: `(x_center = (xmin+xmax)/2, y_center = (ymin+ymax)/2, width = xmax-xmin, height = ymax-ymin)`.
4. Клипируем в [0, 1] на случай погрешностей аннотации за границами кадра.

**Потеря информации**: для сильно вытянутых или невыпуклых полигонов (например, длинный штрих септориоза) описанный прямоугольник существенно больше реальной области симптома. Это увеличивает IoU-overlap между соседними bbox и потенциально снижает точность локализации при обучении. Для высокоточной детекции в будущем имеет смысл:
- обучать на тонких сегментационных масках (YOLO-seg вместо YOLO-det),
- или использовать oriented bounding boxes (OBB) — ориентированные прямоугольники.

В рамках диплома остаёмся с axis-aligned bbox, так как:
- выбранный детектор YOLOv10 работает именно с ними,
- точной сегментации для task planner НИР2 не требовалось,
- масштабируемость проще (OBB и segmentation дают ×2–3 вычислительную стоимость).

## 5. Стратифицированное разбиение 70/20/10

**Цель**: train/val/test должны иметь одинаковое распределение классов. Простое случайное разбиение при сильном дисбалансе и малом объёме редких классов может «вымыть» класс из val или test, делая оценку для него невозможной.

**Алгоритм**:
1. Для каждого изображения определяем **доминантный класс** — класс с наибольшим числом bbox на этом снимке.
2. `sklearn.model_selection.train_test_split` с параметром `stratify=[dominant_class_per_image]`:
   - первый split: `test_size = 0.10` (отделяем test)
   - второй split на оставшихся: `val_size = 0.20 / 0.90 ≈ 0.222` (отделяем val из train+val)
3. Класс с <3 изображениями не стратифицируется (sklearn требует минимум 3 образца на класс в stratify), но у нас после фильтра MIN_IMAGES=100 таких классов нет.

**Почему стратификация по доминантному классу, а не по набору всех классов на кадре**: multilabel-стратификация (`iterstratify`, `MultilabelStratifiedShuffleSplit`) теоретически точнее учитывает со-аннотации, но:
- добавляет зависимость (`iterative-stratification` — отдельный пакет),
- даёт предельное улучшение <2% по редким классам при наших объёмах,
- хуже интерпретируется в логах.

Для нашего случая (в подавляющем большинстве снимков — один-два доминирующих класса) разница несущественна.

**Результат**: train 3 109 / val 889 / test 445 изображений. По аннотациям распределение почти идеально следует заданным пропорциям (±1–2%), что подтверждает корректность стратификации.

## 6. Практические детали реализации

- **Image linking**: исходные JPG подключаются в `dataset/train/images/` и т.д. через **симлинки**, не копируются. Это экономит ~3 ГБ дискового пространства и гарантирует, что любые изменения в исходных файлах отразятся в датасете без повторной конвертации.
- **Label files**: пишутся как обычные файлы (~4–5 КБ каждый), так как содержат уже преобразованные координаты.
- **data.yaml**: формат Ultralytics YOLO — содержит пути train/val/test, число классов `nc`, список имён `names`. Именно этот файл будет передаваться в `YOLO.train(data=...)`.
- **classes.yaml**: дубль маппинга классов для удобства в других скриптах (аугментация, оценка).
- **Валидация**: после записи проверяем, что у каждого изображения есть `.txt`, все class_id в диапазоне, все координаты в [0, 1]. Вывод: **0 ошибок** на 4 443 изображениях × 15 867 bbox.
