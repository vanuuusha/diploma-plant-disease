## 1. Проблема, которую решает этот этап

После task_04 объём train-сплита вырос в 3×, но **пропорции классов сохранились**. Imbalance ratio остался 5.65×: редкие классы по-прежнему представлены втрое меньше, чем доминирующий P2O5.

Почему это плохо для детектора:
- **Loss детектора по умолчанию** (Cross-Entropy для классификации, L1/IoU для локализации) суммируется по всем bbox. При IR=5.65× вклад P2O5 в общий loss в 5.65× больше, чем Заморозков. Градиент всегда «тащит» модель в сторону улучшения P2O5, а Заморозки она фактически игнорирует.
- **Recall на редких классах** остаётся низким даже при высокой accuracy: модель предпочитает перестраховываться и помечать редкий класс как background.
- **Классификатор уходит в prior**: если бы детектор мог выиграть, просто всегда предсказывая «не Заморозки», он бы выиграл — поэтому он так и делает, пока датасет не сбалансирован.

## 2. Методы балансировки: выбор подхода

В литературе и практике есть четыре основных семейства методов:

| Метод | Принцип | Плюсы | Минусы |
|-------|---------|-------|--------|
| **Random undersampling** | Выкинуть часть доминирующих классов | Быстро, просто | Теряются реальные данные; для 8864→1568 потеряли бы 82% информации о P2O5 |
| **Random oversampling** | Дублировать редкие классы без изменений | Сохраняет все данные | Одни и те же примеры → сильное переобучение |
| **Aug-based oversampling** (наш) | Дублировать редкие с агрессивной аугментацией | Разнообразие + сохранение данных | Всё ещё статистически зависимые копии |
| **Generative oversampling** (task_06) | Генерировать синтетические примеры | Максимальное разнообразие | Распределение может не совпадать с реальным |
| **Loss reweighting** (focal/class-balanced loss) | На этапе обучения взвешивать loss обратно пропорционально частоте | Не трогает данные, проще в коде | Не работает так хорошо при сильном дисбалансе (IR >10×); не заменяет разнообразие |

**Выбор**: aug-based oversampling как основной метод (task_05) + generative oversampling для оставшегося дефицита (task_06). Loss reweighting не применяем на уровне данных — оставляем как опцию при финальном обучении (можно добавить `focal_loss=True` в Ultralytics при необходимости).

## 3. Ключевое решение: 70/30 split между task_05 и task_06

**Почему не 100% на aug-oversampling**: если доводить редкие классы до полной цели только через агрессивные копии оригиналов, получаем 10–20 почти-идентичных копий каждого редкого снимка. Модель выучит **артефакты аугментации** (определённые углы поворота, паттерны ElasticTransform) как устойчивые признаки класса — это классический overfitting на синтетическом распределении.

**Почему не 100% на generative**: Stable Diffusion генерирует изображения из априорного распределения LAION-5B, которое **не совпадает** с распределением нашего реального датасета (пшеница специфического сорта, конкретные условия съёмки). 100% генеративной синтетики сдвинуло бы обучение в сторону «обобщённой пшеницы с картинок интернета», а не «нашей тестовой пшеницы».

**70% aug + 30% generative** — компромисс: большая часть дополнений остаётся близкой к исходному распределению (через агрессивную аугментацию реальных снимков), а 30% вносит **семантическое разнообразие** (диффузия реально генерирует новые визуальные композиции, не являющиеся геометрическими копиями).

Это соотношение эмпирическое. В литературе по semi-supervised learning часто рекомендуют 70–80% реальных/15–30% синтетических примеров для домена с ограниченными данными. Наш выбор согласуется с этим диапазоном.

## 4. Формула таргета по классам

```
final_target     = 0.8 × median(annotations per class after task_04)
gap[c]           = max(0, final_target − count_after_aug[c])
task_05_target[c] = count_after_aug[c] + 0.7 × gap[c]
# оставшиеся 30% × gap[c] покрывает task_06
```

**Почему 0.8 медианы, а не медиана/среднее/максимум**:
- **Медиана** (а не среднее) — устойчивее к выбросам (P2O5 = 8864 тянет среднее вверх). Медиана = 3196 (Фузариоз).
- **80% медианы** (2556), а не 100% — полное выравнивание требовало бы добавить классам 2× больше аугментаций, что усилило бы эффект переобучения на них. 80% — «хорошо достаточно», без перебора.

## 5. Агрессивный пайплайн аугментаций

Для oversampling используется более агрессивный набор трансформаций, чем в task_04. Причины:
- Для 4 редких классов у нас и после oversampling будет меньше визуального разнообразия, чем для крупных. Агрессивнее = больше разнообразия из того же материала.
- Эти трансформации **не применяются в val/test** — можно позволить себе их без риска.

### Сравнение с task_04

| Параметр | Task 04 (classic) | Task 05 (aggressive) |
|----------|-------------------|----------------------|
| Affine rotate | ±20° | **±30°** |
| Affine scale | 0.85–1.15 | **0.80–1.20** |
| Affine translate | 10% | **15%** |
| RandomResizedCrop scale | 0.7–1.0 | **0.6–1.0** (сильнее кроп) |
| Brightness/Contrast | ±0.2 | **±0.3** |
| HSV hue/sat/val | 10/20/15 | **15/25/20** |
| **Доп. трансформации** | — | **ElasticTransform (α=30, σ=5)**, **GridDistortion** |

### ElasticTransform и GridDistortion — почему именно они

Это **нелинейные деформации**. Все предыдущие аугментации (Affine, Crop, Flip) — линейные или кусочно-линейные: сохраняют прямые линии. Реальные листья пшеницы **изгибаются, закручиваются, сминаются** — это нелинейная деформация, которая не моделируется линейными трансформациями.

- **ElasticTransform**: применяет случайное гладкое смещение каждого пикселя. Эффект — как будто лист провели через мелкую рябь. Параметр α контролирует амплитуду смещения, σ — гладкость (меньше σ = более «мятый» эффект).
- **GridDistortion**: делит изображение на сетку и смещает узлы сетки случайным образом, интерполируя между ними. Более структурированная деформация, удобно для имитации перспективных искажений.

Эти две трансформации дают качественно новый тип вариативности, недостижимый для `Affine`. В Ultralytics YOLO их нет «из коробки» — поэтому обрабатываем через Albumentations.

## 6. Практические ограничения реализации

### Лимит повторов одного изображения: MAX_REPEAT_PER_IMAGE = 10
При очень большом дефиците и малом пуле seed-снимков скрипт мог бы генерировать 50–100 аугментаций с одного кадра. Это создаёт **семейство статистически зависимых снимков**, на которых модель может переобучиться сильнее, чем на исходнике. Эмпирический порог 10 — за ним прирост разнообразия резко падает.

У нас для Заморозков (самый большой дефицит) на ~440 снимках-носителях нужно +692 bbox ≈ ~410 дополнительных аугментаций. При лимите 10 и среднем 2 bbox/снимок это с запасом: можно набрать нужное количество, не перебирая ни одно изображение чаще 1–2 раз.

### Seed выбирается строго из оригинальных снимков
Не из `_aug1`/`_aug2` копий, созданных в task_04. Почему: это создало бы цепочку «аугментация → аугментация того же снимка снова», эффект — как в компрессии JPEG-поверх-JPEG: артефакты накапливаются.

### Входной датасет: `dataset_augmented/` (не `dataset/`)
Важно: oversampling делается **поверх** уже аугментированного train, а не поверх оригинала. Почему: мы хотим, чтобы в финальном train были И оригиналы, И стандартные аугментации, И балансирующие — всё вместе. Если бы брали только оригинал, пришлось бы ещё раз делать ×3 шаг.

## 7. Val/test: строго без изменений

Как и в task_04, val и test подключены симлинками на исходные (не аугментированные) файлы. **Единственный сплит, который растёт — это train**. Это соответствует стандартной практике ML-экспериментов: test-распределение должно быть как можно ближе к реальному продакшн-распределению.

### Дополнительный (побочный) эффект
Поскольку на некоторых изображениях Заморозков/Септориоза присутствуют также аннотации других классов (co-occurrences), при oversampling этих снимков попутно размножаются и их. Этот эффект **мал** (у крупных классов прирост <5%) и в таблицу не вынесен, но стоит отметить: дисбаланс уменьшается ещё немного за счёт этого.

## 11. Материал для диплома (§ 3.3.2 или § 3.4 «Балансировка»)

**Структура раздела**:

1. **Проблема**: показать imbalance ratio после task_04 и объяснить, почему это вредит обучению детектора (loss-веса, recall редких классов).

2. **Методы балансировки** — таблица сравнения 4 подходов (undersampling / random oversampling / aug oversampling / generative oversampling / loss reweighting) с плюсами и минусами. Это хороший обзорный кусок для главы с теоретической частью.

3. **Выбранный гибрид (70% aug + 30% generative)** — обоснование пропорций:
   - чисто aug-oversampling → переобучение на артефактах аугментации,
   - чисто generative → сдвиг распределения от реального,
   - гибрид 70/30 сохраняет близость к реальному распределению при достаточной вариативности.

4. **Формула и параметры**: таргет 0.8 медианы, aggressive-пайплайн с добавлением ElasticTransform/GridDistortion — объяснить, что эти трансформации дают нелинейные деформации, недостижимые для Affine.

5. **Результат**: imbalance ratio 5.65× → 3.92×, таблица по классам, визуализация `balance_comparison.png`.

6. **Ограничения**: остаётся дефицит ~800 bbox до цели, который закроет task_06 через Diffusion.
