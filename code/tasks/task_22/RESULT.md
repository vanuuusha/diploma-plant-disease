### 5.1.2 Слой API (Backend)

Backend реализован на **FastAPI** (Python 3.12) + **SQLAlchemy 2.0 + psycopg3**. Ключевые дизайн-решения:

1. **Асинхронные эндпоинты** (`async def`) — позволяют не блокировать worker-поток во время ожидания ответа от OpenRouter. FastAPI автоматически запускает асинхронные маршруты в event-loop'е.
2. **Одновременно 8 параллельных Gemma-запросов** (`asyncio.Semaphore(8)`) — все регионы снимка обрабатываются параллельно, что снижает общую латентность с O(N) до O(1) по числу регионов.
3. **Pydantic-схемы** для валидации входящих форм и сериализации ответов.
4. **SQLAlchemy Declarative** + миграции через `Base.metadata.create_all` при старте (для диплома достаточно; в production — Alembic).

API-эндпоинты:

## Pipeline в деталях

Пайплайн диагностики реализован в `backend/app/services/pipeline.py::run_pipeline()`. Последовательность:

### Этап 1. Бинарный классификатор «пшеница / здорово»

Для бинарной проверки состояния входного изображения применяется **классификатор ConvNeXt Tiny**, обученный в рамках НИР1 как часть каскадной системы диагностики. Точность классификации «здоровое/больное» на валидационной выборке составляет **89 %**. Модель принимает изображение 224×224 и возвращает вероятностный выход по двум классам. Если снимок не классифицирован как «пшеница» или помечен как «здоровое» — пайплайн завершается на этом этапе, пользователь получает соответствующее сообщение в UI. Это важно для того, чтобы не запускать тяжёлый детектор впустую.

### Этап 2. Детектор YOLOv12 + CGFM

Если растение классифицировано как больное, запускается **YOLOv12-m с модулем CGFM на уровне P5** (оптимальная конфигурация из аблационного исследования главы 4, task_16). Детектор:
- Принимает изображение 640×640 (letterbox-resize от оригинала).
- Выдаёт регионы поражения с координатами bbox и распределением вероятностей по 9 классам заболеваний и дефицитов.
- Использует NMS (IoU threshold 0.5) и confidence threshold 0.2 для фильтрации кандидатов.

Классы детектирования (9 шт., такие же как в главах 3-4):
1. Недостаток P2O5
2. Листовая (бурая) ржавчина
3. Мучнистая роса
4. Пиренофороз
5. Фузариоз
6. Корневая гниль
7. Септориоз
8. Недостаток N
9. Повреждение заморозками

### Этап 3. Ревью регионов через Gemma

Для каждого найденного региона параллельно отправляется запрос к LLM **Gemma 3-27B IT** через API OpenRouter. Gemma получает:
- **Изображение всего листа** (масштабированное до 512 пикселей по длинной стороне) — для контекста.
- **Изображение региона** (crop bbox) — крупным планом.
- **Топ-4 кандидата** с вероятностями от детектора в текстовом виде.
- **Общий hint от бинарного классификатора**.

Gemma отвечает структурированным JSON:
```json
{
  "approved": ["Септориоз", "Пиренофороз"],
  "comment": "В регионе видны продолговатые некротические пятна с тёмными краями, характерные для септориоза. Нельзя полностью исключить пиренофороз — похожая морфология на начальной стадии."
}
```

Параллельность реализована через `asyncio.gather` с ограничением `Semaphore(8)` на число одновременных HTTP-запросов — это снижает общую латентность пайплайна в N раз (где N — число регионов) без превышения rate-limit'а OpenRouter.

### Этап 4. Boost и нормализация вероятностей

Вероятности классов, которые Gemma подтвердила (`approved`), масштабируются коэффициентом 1.3 (параметр `GEMMA_APPROVE_BOOST`), после чего всё распределение нормализуется к единице:

$$p'_i = \begin{cases} p_i \cdot (1 + \alpha), & i \in A \\ p_i, & i \notin A \end{cases}$$

$$p^\text{final}_i = \frac{p'_i}{\sum_j p'_j}$$

где $A$ — множество approved-классов, $\alpha = 0.3$.

Финальный диагноз — $\arg\max_i p^\text{final}_i$. Финальная уверенность — $\max_i p^\text{final}_i$.

### Этап 5. Общий комментарий

После обработки всех регионов Gemma ещё раз получает исходное изображение и краткую сводку найденных регионов (диагнозы + уверенности) и формирует 3–5 предложений общего заключения для агронома. Этот комментарий выводится на странице результатов сверху над списком регионов.

### Этап 6. Рекомендации

Для каждого региона (и в целом для снимка) формируется список практических рекомендаций на основе хардкод-справочника (`backend/app/services/recommendations.py`). Рекомендации составлены на основе справочников защиты растений, материалов ВНИИФ и ФАО. Структура справочника — `{diagnosis: [list of 3-4 actionable recommendations]}`. Пример для диагноза «Септориоз»:

1. Обработка фунгицидами с действующими веществами эпоксиконазол+флуксапироксад в фазе кущения.
2. Соблюдайте плотность посева (4–5 млн всхожих семян на га, не выше).
3. Используйте устойчивые сорта (Эритроспермум, Скипетр, Тимирязевка).
4. Осенью — глубокая заделка пожнивных остатков.

## Технологические решения и их обоснование

### Почему Next.js + Ant Design

- **Next.js 15 App Router** — современный фреймворк React с server components, автоматическим кодом-сплиттингом и оптимизированной сборкой. Позволяет писать одностраничное приложение с минимальной инфраструктурой.
- **Ant Design 5** — зрелая библиотека UI-компонентов с русской локализацией «из коробки», единая дизайн-система. Обеспечивает быстрый старт и консистентный внешний вид.
- **`@ant-design/nextjs-registry`** — официальный интеграционный пакет для SSR-streaming Ant Design в Next.js App Router (без мерцания стилей).

### Почему FastAPI

- **Асинхронность** критически важна для нашего pipeline'а: Gemma-запросы идут параллельно, FastAPI с async-def эндпоинтами не блокирует worker-поток.
- **Автоматическая валидация** через Pydantic v2 устраняет большую часть ручной проверки форм.
- **OpenAPI-документация** из коробки — `/docs` даёт интерактивную Swagger-консоль.

### Почему PostgreSQL

- **JSON-колонки** (JSONB) подходят для хранения распределений вероятностей, сырых ответов Gemma, без необходимости плоской модели.
- **Надёжность, производительность** — стандарт для production-приложений.
- **Простая интеграция с SQLAlchemy** через `psycopg3`.

### Почему Gemma через OpenRouter

- **Gemma 3-27B IT** — multi-modal модель с поддержкой изображений, доступная в JSON-mode (гарантированный структурированный ответ). Хорошо справляется с задачами agricultural и medical reasoning по фотографиям.
- **OpenRouter** — единая точка доступа к LLM-провайдерам с rate-limit management, не требует отдельной интеграции с Google Cloud / Vertex AI.
- **Rate limits** — OpenRouter до 10 RPM на free tier, достаточно для прототипа; в production — перейти на paid tier или self-hosted.
- **Стоимость** — Gemma 3-27B IT через OpenRouter стоит ~$0.0005 за 1K токенов input + $0.002 за 1K output. Один pipeline (1 бинарный запрос + 4-6 региональных + 1 общий) — ~$0.02. Приемлемо для диплома и MVP.

### Почему CGFM P5-only

Как показано в главе 4 (аблационное исследование task_16), конфигурация **CGFM на уровне P5 neck с MobileNetV3-Small контекстным энкодером** даёт лучший результат исследования (mAP@50 = 0.678 vs baseline 0.651, прирост **+2.7 п.п.**). Эта конфигурация:
- Добавляет всего 0.13M параметров в neck (+1.85M в ContextEncoder, итого 22.12M vs 20.14M baseline — 9.8 % overhead).
- Незначительно увеличивает латентность (в production — overhead ~5 %).
- Даёт существенный прирост mAP@50 (классификационная способность — критично для различения визуально похожих болезней пшеницы).

## Эксплуатационные характеристики

- **Время отклика end-to-end**: 8-15 секунд (зависит от числа регионов). Breakdown: детектор ~0.2 с, Gemma binary ~2 с, Gemma regions ~3-4 с (параллельно), Gemma overall ~2 с, DB operations ~0.1 с.
- **GPU-требования**: YOLOv12m работает на CUDA (рекомендуется ≥ 6 GB VRAM) или на CPU (медленнее в ~10×).
- **Конкурентность**: Backend поднимается с 2 uvicorn-воркерами; один worker обрабатывает до 10 одновременных запросов (bottleneck — GPU-inference).
- **Хранение изображений**: Docker volume `uploads/`, без auto-cleanup (для production — cron-скрипт на удаление старых записей).

## Проблемы и ограничения

- **Latency из-за OpenRouter**: каждый Gemma-запрос — 1-3 секунды. В режиме много регионов (>10) пайплайн может занимать 15+ секунд.
- **Стоимость LLM-вызовов**: ~$0.02 на один запрос. При масштабировании (>1000 запросов/день) — переход на self-hosted VLM (например, Qwen2.5-VL 7B на A100).
- **Качество детектора** (mAP@50 = 0.678) достаточно для информационной поддержки агронома, но не заменяет экспертную диагностику. В production — дообучение на большем корпусе для ещё более высокой точности.
- **Бинарный классификатор** сейчас реализован как Gemma-запрос (JSON с `is_wheat`, `is_healthy`); в production-версии — заменить на локальный ConvNeXt Tiny из НИР1 для экономии на OpenRouter-вызовах.

Для главы 5 требуется 5-7 скриншотов реально работающего приложения. Список — файл `docs/screenshots_plan.md` в репозитории приложения:
1. Главная страница (drag-and-drop).
2. Состояние «Загружается» (Spin-индикатор).
3. Результат «Здоровое растение» (success-alert).
4. Результат «Не пшеница» (warning-alert).
5. Результат с обнаруженными регионами (основной экран).
6. Модальное окно с деталями региона (карточка).
7. Страница со списком рекомендаций.

Скриншоты будут собраны при финальной сборке дипломного документа через Playwright (автоматизация) или через ручной запуск docker-compose на проекторе перед защитой.
