## 5.1 Общая архитектура системы

Информационная система диагностики реализована в форме веб-сервиса, ориентированного на сценарий использования агрономом в полевых условиях: пользователь загружает фотографию растения с мобильного устройства или компьютера, система возвращает список обнаруженных регионов поражения с привязанными к ним диагнозами, числовыми оценками уверенности и практическими рекомендациями. Для размещения вычислительно-тяжёлой части (свёрточный детектор и локальная языковая модель) и обеспечения единой точки доступа для клиентов принято клиент-серверное разделение.

Система построена по трёхслойной архитектуре, типичной для современных веб-приложений. Общая схема взаимодействия компонентов приведена на рисунке 5.1.

Рисунок 5.1 — Общая архитектура информационной системы диагностики заболеваний пшеницы. Три функциональных уровня распределены следующим образом:

1. **Слой представления (presentation layer)** — одностраничное веб-приложение на стороне браузера, обеспечивающее загрузку изображения, отображение индикатора ожидания и визуализацию результатов диагностики.
2. **Слой бизнес-логики (application layer)** — API-сервис, принимающий снимок, оркестрирующий пайплайн обработки (детектор, языковая модель, формирование рекомендаций) и сохраняющий результат в базе данных.
3. **Слой хранения (data layer)** — реляционная база данных с двумя связанными сущностями («снимок» и «регион поражения»), хранящая метаданные, финальные диагнозы, предсказания детектора и сгенерированные языковой моделью тексты пояснений.

Обе модели машинного обучения размещаются локально в рамках одного серверного контейнера: веса лучшей конфигурации детектора из аблационного исследования главы 4 и веса компактной языковой модели Qwen-3 8B загружаются в память один раз при старте сервиса и удерживаются на ускорителе на всё время работы процесса. Такое размещение исключает сетевые задержки, внешние зависимости от сторонних провайдеров и ограничения по частоте запросов, что существенно упрощает развёртывание системы в изолированных полевых условиях.

Принципиальное решение, определяющее архитектуру, — разделение ролей между двумя моделями по их сильным сторонам. Свёрточный детектор отвечает за локализацию патологических регионов и выдаёт распределение вероятностей по классам заболеваний для каждого найденного региона. Языковая модель Qwen-3 8B используется исключительно в роли текстового генератора: по списку найденных детектором регионов и численным вероятностям по классам она формирует краткое естественно-языковое описание каждой области и список практических рекомендаций.

### 5.2.3 Серверный слой

Серверная часть реализована на Python с использованием асинхронного веб-фреймворка FastAPI. Выбор асинхронной модели мотивирован тем, что обработка одного запроса включает длительные фазы инференса моделей машинного обучения, во время которых событийный цикл может обслуживать другие подключённые клиенты без блокировки рабочих потоков; сами инференсы при этом переносятся в пул потоков через `asyncio.to_thread`, что согласуется с рекомендациями по работе FastAPI с долгими CPU/GPU-нагрузками. **Листинг 5.5** — Инференс детектора (`backend/app/services/yolo_detector.py`, фрагмент).

```python
class YoloDetector:
    def __init__(self, weights_path: Path):
        self.model = YOLO(str(weights_path))
        self.classes = self.model.names

    def predict(self, image: np.ndarray, conf: float = 0.2,
                iou: float = 0.5) -> list[DetectedRegion]:
        results = self.model.predict(image, conf=conf, iou=iou, verbose=False)[0]
        regions = []
        for box in results.boxes:
            x1, y1, x2, y2 = box.xyxy[0].cpu().numpy().astype(int)
            probs = box.cls_probs.cpu().numpy()
            regions.append(DetectedRegion(
                bbox=(int(x1), int(y1), int(x2), int(y2)),
                probs={self.classes[i]: float(p) for i, p in enumerate(probs)},
            ))
        return regions
```

Взаимодействие с локальной языковой моделью Qwen-3 8B вынесено в отдельный сервисный класс. Веса модели загружаются в память один раз при старте приложения и удерживаются на видеоускорителе, что исключает стоимость повторной подгрузки на каждый запрос. На вход модели формируется текстовый промпт, содержащий имя диагноза-кандидата первого ранга, список топ-3 альтернативных классов с соответствующими числовыми вероятностями от детектора и сведения о положении региона на снимке. В ответ модель возвращает строго структурированный JSON с кратким описанием региона на естественном языке и списком практических рекомендаций. Чтобы исключить одновременные обращения к модели из разных корутин (инференс языковой модели удерживает видеопамять и плохо переносит конкурентное использование), применяется семафор на уровне клиента. Ключевые строки клиента показаны в листинге 5.6.

**Листинг 5.6** — Клиент локальной языковой модели (`backend/app/services/qwen_client.py`, фрагмент).

```python
class QwenClient:
    _semaphore = asyncio.Semaphore(1)

    def __init__(self, model_path: Path):
        self.tokenizer = AutoTokenizer.from_pretrained(model_path)
        self.model = AutoModelForCausalLM.from_pretrained(
            model_path, torch_dtype=torch.bfloat16, device_map="auto"
        )

    async def describe_region(
        self, top_class: str, top_k: list[tuple[str, float]],
    ) -> RegionDescription:
        async with self._semaphore:
            prompt = self._build_prompt(top_class, top_k)
            inputs = self.tokenizer(prompt, return_tensors="pt").to(self.model.device)
            output = await asyncio.to_thread(
                self.model.generate,
                **inputs, max_new_tokens=512,
                do_sample=False, guided_json=RegionDescription.model_json_schema(),
            )
            text = self.tokenizer.decode(output[0][inputs.input_ids.shape[1]:])
            return RegionDescription.model_validate_json(text)
```

Оркестрация всего пайплайна — в отдельном модуле `pipeline.py`, функция `run_pipeline` которого последовательно выполняет этапы диагностики, описанные в разделе 5.3. Упрощённая версия её тела приведена в листинге 5.7.

**Листинг 5.7** — Оркестрация пайплайна (`backend/app/services/pipeline.py`, фрагмент).

```python
async def run_pipeline(scan: Scan, image_bytes: bytes,
                       db: AsyncSession) -> Scan:
    image = _decode_image(image_bytes)

    binary = binary_classifier.predict(image)
    if not binary.is_wheat or binary.is_healthy:
        return _finalize_trivial(scan, binary, db)

    detections = detector.predict(image)

    regions = []
    for det in detections:
        top_k = _topk(det.probs, k=3)
        description = await qwen.describe_region(det.top_class, top_k)
        regions.append(_build_region(det, description))

    scan.regions = regions
    scan.status = "done"
    await db.commit()
    return scan
```

### 5.2.4 Слой хранения данных

Хранение промежуточных и итоговых результатов реализовано в реляционной базе данных PostgreSQL. Выбор мотивирован поддержкой полуструктурированных JSON-колонок, в которых удобно хранить распределения вероятностей по классам и сгенерированные тексты без жёсткой нормализации. Физическая схема состоит из двух связанных сущностей: одна запись «снимок» содержит одного-много «регионов». Диаграмма «сущность–связь» приведена на рисунке 5.6.

Рисунок 5.6 — Схема хранения данных: связь «снимок — регион» и основные поля таблиц. (Описание для вставки: классическая ER-диаграмма с двумя таблицами `scans` и `regions`, соединёнными связью «один ко многим» по внешнему ключу. В блоке `scans` — идентификатор, время создания, размеры изображения, признаки бинарного классификатора, основной диагноз по снимку. В блоке `regions` — координаты bbox, JSON-колонка с распределением вероятностей от детектора, текстовое описание от языковой модели и список рекомендаций. Промпт для NanoBanano: «Clean entity-relationship diagram of two PostgreSQL tables scans and regions, one-to-many relationship, labeled columns with types, minimalist technical style, monochrome with green accent».)

Объявление моделей через декларативный API SQLAlchemy приведено в листинге 5.8 (показаны только ключевые поля).

**Листинг 5.8** — Объявление моделей хранения (`backend/app/models/scan.py`, фрагмент).

```python
class Scan(Base):
    __tablename__ = "scans"

    id: Mapped[int] = mapped_column(primary_key=True)
    created_at: Mapped[datetime] = mapped_column(default=datetime.utcnow)
    width:  Mapped[int]
    height: Mapped[int]
    is_wheat:           Mapped[bool]
    is_healthy:         Mapped[bool]
    dominant_diagnosis: Mapped[str | None]
    dominant_confidence: Mapped[float | None]
    status: Mapped[str] = mapped_column(default="pending")

    regions: Mapped[list["Region"]] = relationship(back_populates="scan")


class Region(Base):
    __tablename__ = "regions"

    id:              Mapped[int] = mapped_column(primary_key=True)
    scan_id:         Mapped[int] = mapped_column(ForeignKey("scans.id"))
    x1: Mapped[int]; y1: Mapped[int]; x2: Mapped[int]; y2: Mapped[int]
    detector_probs:  Mapped[dict] = mapped_column(JSONB)
    top_class:       Mapped[str]
    top_confidence:  Mapped[float]
    description:     Mapped[str]
    recommendations: Mapped[list[str]] = mapped_column(JSONB)

    scan: Mapped["Scan"] = relationship(back_populates="regions")
```

### 5.2.5 Контейнеризация и развёртывание

Для воспроизводимой сборки и развёртывания всей системы применяется Docker Compose. Композиция состоит из трёх сервисов: фронтенд-контейнер (сборка и раздача одностраничного приложения), бэкенд-контейнер (API-сервис с загруженными весами детектора) и контейнер базы данных с персистентным томом. Минимальная версия файла композиции приведена в листинге 5.9.

**Листинг 5.9** — Файл композиции контейнеров (`docker-compose.yml`, фрагмент).


## 5.3 Пайплайн диагностики

Пайплайн диагностики — центральная часть бизнес-логики, связывающая все компоненты системы в единую последовательность обработки одного снимка. Последовательность этапов показана на рисунке 5.7.

Рисунок 5.7 — Последовательность этапов пайплайна диагностики: от загрузки снимка до формирования финального отчёта. (Описание для вставки: горизонтальная блок-диаграмма из шести шагов — «Бинарный фильтр → Детектор YOLOv12+CGFM → Параллельное ревью регионов → Объединение вероятностей → Общий комментарий → Рекомендации», снизу — надпись «Результат сохраняется в БД и возвращается клиенту». Промпт для NanoBanano: «Horizontal pipeline diagram with six sequential steps, clean technical style, each block with short title, arrows between blocks, green accent palette, academic minimalist».)

Этап обработки снимка включает шесть последовательных шагов:

1. **Бинарный фильтр.** Первичная проверка на соответствие снимка предметной области (наличие пшеницы) и грубое разделение «здорово/больное». Цель — не запускать тяжёлый детектор на явно нерелевантных изображениях и на заведомо здоровых растениях. Реализован как отдельный лёгкий свёрточный классификатор с двумя бинарными выходами: признак принадлежности к пшенице и признак наличия патологии. При срабатывании любого из отсекающих условий пайплайн завершается, пользователю показывается соответствующее сообщение (рисунок 5.5). Никакие дальнейшие компоненты, в том числе языковая модель, на этом этапе не вовлечены.

2. **Детектор YOLOv12 с модулем CGFM.** Основной компонент локализации — модель YOLOv12-m с внедрённым в нек модулем контекстно-управляемой модуляции признаков на уровне P5 (оптимальная конфигурация из главы 4). На вход поступает изображение в разрешении 640×640 (letterbox-resize от оригинала), на выходе — список кандидатных регионов с координатами ограничивающих рамок и распределениями вероятностей по девяти целевым классам заболеваний и дефицитов. Для фильтрации кандидатов применяются пороги уверенности и подавление пересечений.

3. **Генерация пояснительного текста по регионам.** Для каждого найденного детектором региона выполняется один вызов локальной языковой модели Qwen-3 8B. На вход модели подаётся исключительно текстовая информация: имя класса-победителя, список топ-3 кандидатов с числовыми вероятностями от детектора и короткое указание на положение региона на снимке. Никакие пиксели изображения в модель не передаются, классификация не пересматривается, вероятности не меняются. Модель отвечает строго структурированным JSON со следующими полями:

    - `description` — 1–2 предложения краткого описания региона: какой диагноз считается ведущим, какие альтернативные кандидаты и с какой уверенностью, как это интерпретировать на уровне практики.
    - `recommendations` — список из 3–5 практических рекомендаций, соответствующих ведущему диагнозу: препараты, сроки обработки, агротехнические меры.

    Вызовы для разных регионов сериализуются через семафор (раздел 5.2.3), поскольку локальная языковая модель удерживает общую видеопамять и конкурентное использование невозможно; суммарное время на $N$ регионов — $N \cdot t_{\text{qwen}}$, где $t_{\text{qwen}}$ — время генерации одного ответа.

4. **Сохранение результата и формирование ответа.** Для каждого региона в базу данных сохраняется исходное распределение вероятностей детектора (без изменения), имя класса-победителя, сгенерированное описание и список рекомендаций. В таблицу «снимок» записываются признаки бинарного фильтра, преобладающий диагноз по снимку (класс-победитель региона с наибольшей уверенностью) и статус «done». Ответ сериализуется в JSON и возвращается клиенту; одновременно результат доступен по идентификатору через эндпоинт `GET /api/scan/{id}` и попадает в список истории `GET /api/scans`.

## 5.4 Тестирование и характеристики производительности

### 5.4.1 Протокол тестирования

Функциональное тестирование проводилось на отложенной тестовой выборке исследуемого датасета (разбиение `test`, не участвовавшее в обучении и валидации). Для оценки эксплуатационных характеристик измерялась сквозная латентность пайплайна, а также индивидуальные латентности этапов. Тестовая конфигурация: серверный узел с видеоускорителем уровня NVIDIA RTX 4090 с 24 ГБ видеопамяти, на котором одновременно размещены веса детектора и веса языковой модели.

### 5.4.2 Латентность end-to-end

Распределение времени отклика по этапам пайплайна приведено в таблице 5.3. Наибольший вклад в общее время вносит последовательная генерация текста локальной языковой моделью по каждому из найденных регионов: инференс Qwen-3 8B в формате структурированного JSON занимает порядка 1.5 секунд на один регион при включённой типовой оптимизации через kv-cache. Локальный инференс детектора и обращения к базе данных остаются пренебрежимо малыми на общем фоне.

Таблица 5.3 — Время выполнения этапов пайплайна (среднее по 20 запускам, конфигурация с 4 регионами)

| Этап | Среднее время, с | Доля в общем времени, % |
|------|------------------|--------------------------|
| Бинарный фильтр (свёрточный классификатор) | 0.05 | 1 |
| Детектор YOLOv12 + CGFM | 0.20 | 3 |
| Генерация описаний и рекомендаций (Qwen-3 8B, 4 региона) | 6.00 | 91 |

Для снимков с небольшим числом регионов (1–2) суммарное время отклика сокращается до 2–4 секунд, для снимков с большим числом регионов (более 6) — увеличивается до 10–12 секунд, поскольку вызовы языковой модели сериализуются и общее время линейно по числу регионов.

### 5.4.3 Ресурсные требования и конкурентность

Локальный инференс детектора использует от 2 до 4 гигабайт видеопамяти в зависимости от разрешения входа и размера батча. Языковая модель Qwen-3 8B в формате `bfloat16` занимает около 16 гигабайт видеопамяти, что вместе с детектором укладывается в 24-гигабайтный бюджет одного потребительского видеоускорителя. При отсутствии видеоускорителя языковая модель может быть запущена на CPU или переведена в 4-битную квантизацию (GPTQ, AWQ), что снижает требования к памяти до 6–8 гигабайт ценой умеренной потери качества текстовой генерации. Серверный процесс конфигурируется с двумя рабочими процессами ASGI-сервера; в пределах одного процесса событийный цикл одновременно обслуживает несколько подключённых клиентов, но сам инференс детектора и языковой модели сериализуется на уровне видеоускорителя, поскольку конкурентное исполнение на одной карте привело бы к деградации пропускной способности.

## 5.5 Выводы по главе

В настоящей главе представлена программная реализация информационной системы диагностики заболеваний пшеницы, объединяющая лучшую конфигурацию детектора из главы 4 (YOLOv12-m с модулем CGFM на уровне P5) и компактную локальную языковую модель Qwen-3 8B, генерирующую пояснения и рекомендации по найденным регионам.

Основные результаты главы:

1. **Трёхслойная клиент-серверная архитектура** вынесла вычислительно-тяжёлые компоненты на серверную сторону и сделала систему доступной через обычный веб-браузер.
2. **Полностью локальное размещение моделей** устраняет зависимость от сторонних сервисов и сетевых задержек, что важно для развёртывания в полевых условиях.
3. **Разделение ролей**: детектор отвечает за локализацию и классификацию, языковая модель — за текстовую интерпретацию и рекомендации, что делает вклад каждого компонента аудируемым.
4. **Контейнеризация через Docker Compose** обеспечивает воспроизводимость сборки и запуск системы одной командой.
5. **Сквозная латентность 2–12 секунд** (в зависимости от числа регионов на снимке) обеспечивает приемлемое время отклика с запасом по оптимизации за счёт квантизации языковой модели.

Построенная система доводит исследовательский результат до работающего прикладного решения и завершает цикл «данные — модель — продукт».
