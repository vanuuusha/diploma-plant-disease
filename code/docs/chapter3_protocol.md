# Протокол экспериментов главы 3: сравнение детекторов × вариантов датасета

Документ описывает единый экспериментальный протокол для задач `task_07`–`task_11`. Все TASK.md этой группы ссылаются сюда, чтобы исключить расхождения в гиперпараметрах, путях и метриках между детекторами.

---

## 1. Цель блока задач

Количественно оценить, как каждая стадия подготовки данных (исходный датасет → геометрическая аугментация → oversampling → добавление генеративной диффузии) влияет на качество четырёх современных архитектур детекции:
- YOLOv12 (`task_07`)
- RT-DETR (`task_08`)
- Faster R-CNN (`task_09`)
- DETR (`task_10`)

Итоговая сводка — `task_11`: кросс-сравнение 4 × 4, выбор базовой архитектуры для главы 4 (CGFM).

---

## 2. Исходные данные — 4 варианта датасета

Все пути — абсолютные на Linux-сервере (`/home/vanusha/diplom/diploma-plant-disease/code/data/`). Формат — YOLO (images + labels), 9 классов, одинаковые val и test для всех вариантов.

| Тег варианта | Директория | data.yaml | Назначение |
|---|---|---|---|
| `baseline` | `dataset/` | `dataset/data.yaml` | Исходный датасет после конвертации Label Studio → YOLO |
| `aug_geom` | `dataset_augmented/` | `dataset_augmented/data.yaml` | Baseline + классическая геометрическая/фотометрическая аугментация (×3) |
| `aug_oversample` | `dataset_balanced/` | `dataset_balanced/data.yaml` | `aug_geom` + oversampling редких классов |
| `aug_diffusion` | `dataset_final/` | `dataset_final/data.yaml` | `aug_oversample` + синтетика Stable Diffusion (img2img, strength=0.40) |

**Важно**: val и test идентичны между всеми вариантами — различается только train. Это гарантирует, что метрики на test можно напрямую сравнивать.

Классы (id 0–8): Недостаток P2O5, Листовая (бурая) ржавчина, Мучнистая роса, Пиренофороз, Фузариоз, Корневая гниль, Септориоз, Недостаток N, Повреждение заморозками.

## 3. Унифицированные гиперпараметры

Эти значения фиксированы для всех 16 запусков (4 детектора × 4 датасета).

| Параметр | Значение | Комментарий |
|---|---|---|
| `epochs` | 100 | Максимум эпох |
| `patience` | 15 | Early stopping: обучение прерывается, если val-метрика не улучшается 15 эпох подряд |
| `imgsz` | 640 | Размер входа, квадратный |
| `batch` | подбирается | См. §3.1 — зависит от детектора и памяти GPU |
| `optimizer` | default для фреймворка | Ultralytics: AdamW с автоподбором lr; torchvision: SGD momentum=0.9 |
| `lr0` | default фреймворка | Не менять вручную, чтобы не вводить дополнительную переменную |
| `seed` | 42 | Фиксированный |
| `device` | `cuda:0` | RTX 5070 Ti |
| `workers` | 8 | DataLoader workers |
| `pretrained` | True | Все детекторы инициализируются весами на COCO |

### 3.1 Размеры батчей по детекторам

Батч подбирается максимально возможным в рамках 16 ГБ VRAM. Ориентиры:
- YOLOv12-m: `batch=16`
- RT-DETR-l: `batch=8`
- Faster R-CNN (ResNet-50 FPN): `batch=4`
- DETR (ResNet-50): `batch=4`

Если OOM — уменьшить на 2 и перезапустить. Зафиксировать фактический `batch` в `RESULT.md`.

### 3.2 Встроенные аугментации Ultralytics — отключить

Для YOLO и RT-DETR ультралитиковский фреймворк по умолчанию делает mosaic/mixup/HSV-сдвиги «на лету». Для чистоты сравнения **все встроенные аугментации должны быть отключены**, так как сравниваемые стадии подготовки данных уже несут всю запланированную аугментацию. Конкретно в `.train(...)` передавать:

```python
hsv_h=0.0, hsv_s=0.0, hsv_v=0.0,
degrees=0.0, translate=0.0, scale=0.0, shear=0.0, perspective=0.0,
flipud=0.0, fliplr=0.0, mosaic=0.0, mixup=0.0, copy_paste=0.0
```

Для torchvision-детекторов (Faster R-CNN, DETR) — отсутствие встроенных аугментаций настроек не требует; загрузчик должен возвращать изображения без случайных преобразований, только resize и нормализация.

## 4. Единый лог-контракт: что сохранять на каждом запуске

Каждый из 16 прогонов обучения производит фиксированный набор артефактов. Все пути — относительно корня проекта.

### 4.1 Директория результатов

```
code/results/task_NN/<detector>_<variant>/
    metrics.csv                # Покомпонентные метрики по эпохам
    learning_curves.png        # train_loss, val_loss, val_mAP50, val_mAP5095 vs epoch
    confusion_matrix.png       # На test, нормализованная
    per_class_map.csv          # mAP@50 и mAP@50-95 по каждому классу на test
    predictions_examples/      # 6–10 изображений с предсказаниями (на test)
        img_001.jpg
        ...
    fps_measurement.json       # Замер латентности (см. §4.5)
    best.pt                    # Лучшие веса по val-метрике (или иная форма сохранения)
    train.log                  # Лог обучения
```

Имена `<detector>` и `<variant>` из §2: `yolov12`, `rtdetr`, `faster_rcnn`, `detr` × `baseline`, `aug_geom`, `aug_oversample`, `aug_diffusion`.

### 4.2 metrics.csv

Колонки (именно в этом порядке):

```
epoch, train_loss, val_loss, val_precision, val_recall, val_mAP50, val_mAP5095, lr, time_sec
```

Одна строка на эпоху. Первая строка — заголовок. Разделитель — запятая.

### 4.3 learning_curves.png

Одна фигура, 2×2 сабплота:
- train_loss + val_loss (если фреймворк разделяет — вывести обе, иначе одну)
- val_precision, val_recall на одной оси
- val_mAP@50
- val_mAP@50-95

По X — эпоха, по Y — значение. Сетка включена. Размер: 12×8 дюймов, DPI=120.

### 4.4 confusion_matrix.png и per_class_map.csv

Считаются на test после окончания обучения, используя best.pt. Матрица — квадратная 9×9 (строки — true, колонки — pred), нормализация по строкам. `per_class_map.csv`:

```
class_id, class_name, n_test_instances, mAP50, mAP50_95
```

### 4.5 fps_measurement.json

Процедура замера скорости инференса:
1. Прогреть модель: 20 прогонов на случайном батче.
2. Замерить 100 прогонов на одиночных изображениях 640×640 (`batch=1`), `torch.cuda.synchronize()` до и после.
3. Записать: `mean_ms`, `std_ms`, `median_ms`, `p95_ms`, `fps = 1000/mean_ms`, `device_name`, `batch=1`, `imgsz=640`, `precision=fp32`.

Формат JSON:

```json
{
  "detector": "yolov12",
  "variant": "aug_diffusion",
  "imgsz": 640,
  "batch": 1,
  "precision": "fp32",
  "device_name": "NVIDIA GeForce RTX 5070 Ti",
  "warmup_runs": 20,
  "measure_runs": 100,
  "mean_ms": 18.4,
  "std_ms": 0.9,
  "median_ms": 18.1,
  "p95_ms": 19.8,
  "fps": 54.3
}
```

### 4.6 predictions_examples/

6–10 изображений из test с нарисованными bbox (true — зелёные тонкие, pred — красные, с confidence). Желательно показать разнообразие: простые случаи, сложные случаи (overlap, мелкий объект, множественные классы на кадре), 1 явную ошибку. Подборка должна быть одной и той же для всех 16 прогонов, чтобы качественное сравнение было прямым — брать по фиксированному списку имён файлов. Перед началом экспериментов отобрать 10 изображений из `dataset/test/images` и закоммитить их в `code/docs/chapter3_qualitative_sample.txt` (одно имя файла в строке).

## 5. Формат Jupyter-ноутбуков

Каждый детектор — один персистентный ноутбук `code/notebooks/chapter3_<detector>.ipynb`. Ноутбук разделён на разделы; каждый раздел — полный цикл обучения на одном варианте датасета.

### 5.1 Структура ячеек

```
[md]   # Глава 3 — <Имя детектора>
[md]   ## 0. Импорты и конфигурация
[code] # импорты, фиксация seed, проверка GPU, выбор путей
[md]   ## 1. Общие утилиты
[code] # функции логирования, построения графиков, подсчёта FPS, рендеринга предсказаний
[md]   ## 2. Вариант baseline — dataset/
[code] # запуск обучения
[code] # тест + артефакты (confusion matrix, per-class mAP, FPS, qualitative)
[md]   ## 3. Вариант aug_geom — dataset_augmented/
... (аналогично)
[md]   ## 4. Вариант aug_oversample — dataset_balanced/
... (аналогично)
[md]   ## 5. Вариант aug_diffusion — dataset_final/
... (аналогично)
[md]   ## 6. Сводная таблица по 4 вариантам
[code] # DataFrame с mAP50, mAP5095, Precision, Recall, FPS по 4 запускам
[code] # экспорт в code/results/task_NN/summary.csv
```

### 5.2 Требования к ноутбуку

**Персистентность.** Ноутбук не должен очищаться между запусками вариантов — каждая ячейка обучения оставляет выходы (логи, графики) в ipynb-файле. Это позволит просматривать прогресс в браузере во время обучения и после.

**Git-коммиты.** После завершения каждого варианта коммит `chapter3_<detector>: <variant> done` с закоммиченными: ноутбуком (с выводами), директорией результатов `code/results/task_NN/<detector>_<variant>/` (без `best.pt` — веса в `.gitignore`).

**Наблюдаемость.** В начале каждого варианта — вывод текущей эпохи / метрики раз в 5 эпох (`verbose=True` или аналог), чтобы пользователь видел прогресс, открывая ноутбук в JupyterLab в браузере.

## 8. Порядок выполнения задач

Рекомендуемый порядок (зависит от времени обучения, но не от логики):

```
task_07 (YOLOv12)       ← быстрее всего, 4 прогона ≈ 8–12 часов
task_08 (RT-DETR)       ← ≈ 12–16 часов
task_09 (Faster R-CNN)  ← ≈ 16–24 часа
task_10 (DETR)          ← ≈ 20–30 часов (сходимость медленная)
task_11 (сводка)        ← анализ после завершения первых четырёх
```

Можно выполнять параллельно, если есть несколько GPU, но для фиксации одинаковых условий сравнения (температурные режимы карты и т.д.) предпочтительнее последовательный прогон на одной карте.
