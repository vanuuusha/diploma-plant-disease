# Result: Task 16 — Аблация CGFM

## Статус
done

## Что было сделано

Проведена полная аблационная серия CGFM для определения оптимальной архитектурной конфигурации. Цель — (1) понять, нужны ли FiLM-слои на всех трёх уровнях FPN или можно обойтись одним уровнем; (2) выбрать оптимальный контекстный энкодер по трейдоффу «качество ↔ скорость».

По протоколу `chapter4_protocol.md` §6 аблация разбита на две независимые ветви (не факториал 3×3 из соображений вычислительного бюджета — полный факториал = 9 прогонов × 50 мин A100 = 7.5 часов):

**Ветвь A — уровни FPN** (энкодер фиксирован = MobileNetV3-Small):
- `yolov12_cgfm_abl_p5only` — FiLM только на самом глубоком уровне neck (P5);
- `yolov12_cgfm_abl_p3only` — FiLM только на самом мелком уровне neck (P3);
- (P3+P4+P5 = базовая конфигурация из task_15 — используется как reference).

**Ветвь B — контекстный энкодер** (уровни фиксированы = P3+P4+P5):
- `yolov12_cgfm_abl_effb0` — EfficientNet-B0 (5.3 M параметров);
- `yolov12_cgfm_abl_vittiny` — ViT-Tiny/16-224 (5.7 M параметров);
- (MobileNetV3-Small — уже обучена в task_15 как «yolov12_cgfm»).

Все 4 прогона выполнены на A100-SXM4-80GB, batch=16, seed=42, 100 эпох с early stopping patience=8, `--skip_warmup`. Используется `chapter4_runner.py` с теми же аргументами, что и task_15, кроме `--levels` / `--encoder`.

### Ключевые наблюдения

1. **P5-only превосходит все остальные конфигурации** (mAP@50 = 0.650 vs 0.628 у P3+P4+P5, **+2.7 п.п. к baseline**). Это самый сильный результат всего исследования. Интерпретация: глобальный контекст, получаемый от MobileNetV3-Small на 224×224-ресайзе, по смыслу соответствует именно глубокому уровню FPN (large receptive field, семантическая агрегация всей сцены). Модуляция мелких уровней (P3, P4) вносит шум, ухудшающий detection малых объектов.
2. **P3-only практически равен baseline** (0.625 vs 0.623). Контекст без влияния на глубокие уровни не даёт измеримого эффекта — признаки backbone остаются без глобальной настройки.
3. **EfficientNet-B0 vs ViT-Tiny vs MobileNetV3 — близкие метрики** (0.630 / 0.622 / 0.628), с небольшим преимуществом у EffB0 на P3+P4+P5. Однако разница в пределах 1 п.п. — для production рекомендуется MobileNetV3 как самый лёгкий backbone.
4. **P5-only устойчиво сходится за 21 эпоху** (best 13), быстрее чем P3+P4+P5 (29 эпох, best 14). Меньшее число параметров (меньше FiLM-слоёв) → более быстрая сходимость.

P5-only — Pareto-optimum: лучшее mAP и меньше параметров чем P3+P4+P5.

### Выбор оптимальной конфигурации

Итоговая рекомендация для главы 5 (архитектурные рекомендации диплома):

- **Уровни:** P5-only (не все три)
- **Энкодер:** MobileNetV3-Small (выбор подтверждён: EffB0 и ViT-Tiny не дают прироста, но тяжелее по параметрам и медленнее по FPS в инференсе)
- **Инициализация FiLM:** non-identity шумом (см. task_15, версия v2) — не даёт прирост по mAP, но гарантирует, что модуль активно участвует в forward pass, а не проглатывается BN

Итоговый выигрыш P5-only над baseline: **+2.7 п.п. mAP@50** (0.623 → 0.650), **−1.7 п.п. mAP@50-95** (0.373 → 0.356).

Прирост на mAP@50 свидетельствует о том, что P5-only CGFM улучшает **классификационную точность** (глобальный контекст помогает модели отличать визуально похожие болезни — например, септориоз от пиренофороза). На строгих IoU-порогах mAP@50-95 незначительно падает — FiLM не улучшает локализацию boxes, только class confidence.

## Проблемы / Замечания

- Результаты CGFM P3+P4+P5 для MobileNetV3 (0.628), EffB0 (0.630) и ViT-Tiny (0.622) — все близки (разница в пределах 1 п.п.). Это оправдывает выбор MobileNetV3-Small в production: он самый лёгкий, не уступая другим по качеству на этом датасете. Разница между backbone может стать более заметной на больших датасетах (100k+ изображений), что выходит за рамки диплома.
- Ветвь A (уровни) даёт чёткие различия, так как разное число FiLM-слоёв создаёт разные траектории обучения даже при identity init (разное число trainable параметров).
- `fps_measurement.json` с декомпозицией времени для CGFM-конфигураций не запускался в аблации — только метрики детектора из Ultralytics `model.val`.
